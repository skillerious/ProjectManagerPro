Project Manager At a Glance

Understand the core architecture quickly, then jump to the area you need.

Core Product Areas

Project Manager is structured around a workspace-first model: Projects, Templates, Recent activity, Git operations, Extensions, and a deeply configurable Settings system. Every major area is available from the left navigation and supports keyboard-driven workflows for speed.

Who This Documentation Is For

This guide is designed for solo developers, team leads, and power users who need reliable project scaffolding, controlled Git operations, and predictable settings behavior across long-running workflows.

Workspace Root Single source of truth All project actions resolve from it
GitHub Upload Guided dialog flow Progress + confirmation UX
Settings Engine Persistent model Unsaved-change protection built in
Extensions Install + configure Marketplace + local settings

Recommended Startup Checklist

Set your workspace path, confirm your default editor and terminal, configure Git identity and default branch, then verify extension auto-update policy. This baseline removes avoidable setup friction for every new project.

Reading Order

For new users: Quickstart → Workflows → Settings → Git. For maintainers: Architecture → Troubleshooting → FAQ.

Quickstart (First 30 Minutes)

A reliable onboarding sequence to reach a stable daily workflow quickly.

Goal

By the end of this checklist, you will have predictable project creation, consistent Git identity, a configured terminal/editor path, and a clean extension baseline.

1

Set Workspace Root

Open Settings → General and set a default project location. This keeps project discovery, templates, and recent history aligned.

2

Configure Editor + Terminal

Set your preferred editor/terminal paths in Settings so project-open and terminal actions always launch the expected tools.

3

Define Git Identity

Under Settings → Git, set username/email and default branch. This avoids commit and branch drift across repos.

4

Create a Test Project

Create a small project, run terminal commands, and perform one commit cycle to validate your full setup path.

Startup Checklist

  • Workspace path points to your real project root.
  • Open-in-editor behavior matches your preference.
  • Terminal opens in project context.
  • Git executable path is valid.
  • Default branch is explicit (`main` recommended).

Validation Commands

Run in integrated terminal
git --version
node --version
npm --version

If these commands fail, verify terminal path, environment, and tool installation before proceeding.

Solid Baseline

Once Quickstart is complete, most day-to-day issues shift from environment setup to project-specific logic, making diagnostics much faster.

Professional Daily Workflows

Recommended paths for creating, opening, and shipping work without context switching.

1

Set Workspace and Defaults

Open Settings → General and set your default project path, recent-project limit, and launch preferences. This avoids repetitive setup every session.

2

Create from Template or Import Existing Code

Use New Project for generated scaffolds or Import Project for established repositories. Enable automatic Git init and editor launch when required.

3

Run and Validate from Integrated Tools

Use the Tools menu for terminal and dependency installation. Keep workflow friction low by launching directly into your configured terminal context.

4

Version and Publish with Git View

Stage, commit, push, and upload to GitHub using the guided dialogs. For team usage, keep commit identity and default branch policy configured in Settings.

Workflow Tip

Power users typically combine Ctrl+N (new project), Ctrl+` (terminal), and Ctrl+Shift+P (command palette) to keep hands on keyboard and reduce navigation latency.

Team Workflow Baseline

For team consistency, standardize template usage, enforce a shared branch naming policy, and ensure everyone uses the same Git username/email conventions before creating repositories from Project Manager.

Do This

  • Start each workday by selecting the target project explicitly.
  • Run dependency install/refresh before first commit if lockfiles changed.
  • Use smaller commits with clear intent to simplify rollback.

Avoid This

  • Mixing unrelated changes into one commit.
  • Pushing without reviewing staged files and branch target.
  • Leaving terminal/editor paths unvalidated across machines.

Smart Workflow Loop

Run work in short loops: select project → implement → validate → commit → sync. If a step fails, roll back one step instead of restarting the full flow.

Settings System and Persistence

How to configure Project Manager safely with reliable save behavior and unsaved-change protection.

Persistence Model

Settings are normalized before save, validated by schema constraints, and persisted through IPC to local application data. The interface tracks dirty state and prevents accidental loss when navigating away or closing the application.

Save Confirmation Animated dialog Explicit success state with primary action
Unsaved Protection Leave/close prompt Save, discard, or cancel navigation

Recommended Defaults

For stable development flow, keep Open in VS Code enabled, set a sensible Recent Projects Limit (10-25), and define a consistent Default Branch under Git settings to avoid branch naming drift across projects.

Settings Import/Export Strategy

Use settings export as a baseline profile for new machines. Import settings after installation, then run a quick audit of paths (workspace, editor, terminal, and Git executable) to avoid environment-specific drift.

Path Audit Required

After importing settings from another machine, always validate local file paths before running build, terminal, or Git actions.

Git and GitHub Operations

Source control best practices inside Project Manager's Git view and menu actions.

Repository Operations

Use Git view for status inspection, selective staging, commit creation, and remote operations. The interface is tuned for safe sequencing: inspect changes, stage intentionally, then commit and push.

Commit safety checks Pull/push/sync actions Undo last operation Commit history access

GitHub Upload Dialog Pattern

The upload flow uses explicit confirmations and visual progress states. This same design language is now applied across critical confirmations to reduce ambiguity and improve confidence in irreversible actions.

Selective Upload Scope

The GitHub upload modal now includes a file/folder selection tree with filtering and sorting. This lets you publish only the intended project surface and avoid uploading local-only assets, generated output, or confidential files.

Pre-Push Checklist

  • Working tree reviewed (staged + unstaged).
  • Commit message explains intent and scope.
  • Branch target and remote are confirmed.
  • Large/generated artifacts excluded.

Merge Conflict Playbook

  • Open Conflict Assistant to inspect unresolved files.
  • Resolve with Ours/Theirs/Manual strategy as appropriate.
  • Re-run status and continue merge only after clean state.

Safe Upload Sequence

Use the guided Git view in this order: review changes → apply exclusions → confirm selected scope → commit → push. This keeps uploads intentional and repeatable.

Exclusion Rules First

Always exclude generated or machine-local content such as node_modules, dist, caches, logs, and temporary build output before upload. Verify the final selected-item count before publishing.

Never Publish Secrets

Do not upload .env, API keys, certificates, or private tokens. If sensitive data was committed accidentally, rotate credentials immediately and rewrite history before continuing.

Extension Governance

In production teams, keep a lightweight extension policy: approve a small core set, review update frequency, and periodically remove stale tools to maintain startup performance and predictable behavior.

Performance Hygiene

Prune unused extensions regularly and prefer a compact, high-value set. Fewer active extensions mean faster startup and less UI noise.

Architecture and Runtime Model

How the app is structured for stability, secure IPC boundaries, and performance under frequent UI updates.

Core Separation

The application separates privileged operations in the Main process from UI behavior in the Renderer, bridged through a strict preload allowlist. This minimizes unsafe surface area while preserving workflow speed.

Main Services Update / Queue / Workspace Isolated modules with focused responsibilities
Preload Bridge Allowlisted IPC only Blocked channels fail fast during development
Renderer Caches TTL + in-flight dedupe Prevents redundant work during rapid interactions

Main Process Modules

  • UpdateManager: release checks, channels, rollback flow.
  • WorkspaceServices: snapshots, task profiles, indexed search.
  • OperationQueue: serialized long-running operations with retries.
  • ProjectDiscoveryService: cached project scanning and metadata.
  • RendererFileService: safe import/export/path validation boundaries.

Renderer Stability Patterns

  • Chunked rendering for large project lists.
  • Stale request token checks for async view transitions.
  • Search index fallback with cached IPC results.
  • UI state guards for dialogs and navigation history.
  • Event-driven refresh for extension lifecycle updates.

Operational Rule

Any new `ipcRenderer.invoke/on` usage must be mirrored in preload allowlists and IPC contract tests. This prevents runtime breakage and preserves secure-by-default behavior.

Troubleshooting and Recovery

Resolve common issues quickly using built-in diagnostics and safe reset points.

Fast Diagnosis Sequence

  • Confirm target project selection and workspace path.
  • Refresh active view data (Projects/Git/Extensions).
  • Validate tool paths in Settings (Git, terminal, editor).
  • Retry operation with minimal scope and watch status feedback.

Project Not Opening as Expected

Check Settings → General → Open projects in VS Code automatically. If disabled, Project Manager will open paths in the file explorer instead of the editor.

Terminal Launches in the Wrong Directory

Review Settings → Terminal → Terminal working directory behavior. You can force terminal sessions to launch in the project root or another preferred path strategy.

Git Commands Fail

Set a valid Git executable path under Settings → Git. If needed, verify username, email, and default branch values before retrying repository actions.

Upload Did Not Match Expected Files

Open the GitHub upload modal and verify the Upload Scope tree before submitting. Use search and sort controls, then confirm the selected-item summary to ensure the correct folders and files are staged.

Recovery Playbook

If behavior seems incorrect, refresh project list, reopen the target project, run a manual Git status refresh, and retry. If the issue persists, export diagnostics and file a report from the Help menu.

Diagnose Before Resetting

Capture diagnostics first, then change one variable at a time. This avoids masking root causes and gives you clean evidence if you need to report an issue.

Frequently Asked Questions

Short, direct answers to common operational and configuration questions.

Licensing Transparency

This app is a project manager first. The repository uses MIT for source licensing, while the in-app product key flow is a separate personal cryptography experiment included in this project.

Why does project discovery not show a newly created folder immediately?

Project discovery uses cache windows for performance. Trigger a manual refresh in Projects view or switch views to invalidate and rescan.

What is the safest first-time Git configuration?

Set Git executable path, username/email, and default branch in Settings before creating repositories. This prevents identity and branch inconsistencies.

What is the purpose of the product key system in this app?

It is a side experiment for learning cryptography-style implementation details (key generation, validation, secure storage, integrity checks, and device binding). The app itself remains a project manager, not a learning tool.

Are product keys a commercial license that users must buy?

No. This is a personal project, and the included keys are free and provided only because the key system is part of my experimentation. It is not a paid licensing or monetization system.

What license applies to the source code itself?

The repository code is MIT licensed. Anyone can use it under MIT terms, and the in-app key gate does not replace or override those legal source-code licensing terms.

How should teams manage extensions for consistent behavior?

Use a small approved baseline, pin high-impact tools where possible, and review stale/unused extensions monthly to reduce startup overhead.

Does this app support safe rollback after an update channel change?

Yes. Update controls expose channel selection and rollback logic. Always review release notes before switching to prerelease channels.

How do I migrate settings to another machine reliably?

Export settings from the current machine, import on the target machine, then verify environment-specific paths (workspace, editor, terminal, Git).

What should I do if integrated terminal commands fail?

Validate terminal executable, shell availability, and working-directory behavior in Settings. Then rerun from a known-good project path.

When should I use the operation queue?

Use it for long-running operations (clone/export/upload) when you need visibility, retry support, and deterministic execution order.

How can I open this documentation quickly?

Press F1, open Help → Documentation, or use Command Palette and search for documentation actions.
